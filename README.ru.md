# bench-runtime-windows-vs-wsl

Набор бенчмарков для сравнения производительности Windows и WSL.

Тесты показывают, насколько сильно влияет выбор расположения файлов при работе с WSL. Даже в элементарных сценариях
можно получить деградацию производительности в десятки и сотни раз. Это напрямую отражается на скорости выполнения
задач, отзывчивости инструментов разработки и увеличивает нагрузку на SSD.

Важно понимать, что это не баг и не проблема конфигурации WSL. В большинстве случаев причина — неверная модель работы с
файловой системой. Особенно часто это проявляется при использовании Docker на Windows: подключение volume из
Windows-файловой системы (`C:\` → `/mnt/c`) приводит к резкому снижению производительности. Подобное поведение часто
ошибочно интерпретируют как зависание IDE или нестабильность окружения, хотя на деле это ограничение архитектуры.

Типичные примеры смешанных сценариев:

- Проект открыт в IDE из `C:\`, но команды (`npm`, `composer`, `docker`) выполняются внутри WSL
- Docker-контейнеры в WSL используют volume, смонтированный из `C:\` (`/mnt/c/`)
- `node_modules` или `vendor` находятся на Windows FS, а сборка запускается в WSL
- Git-репозиторий лежит в Windows, но операции (`git status`, `checkout`) выполняются из WSL
- База данных в контейнере (WSL), а данные (volume) лежат на Windows FS
- Watcher'ы (webpack, vite) следят за файлами в `C:\`, но выполняются в WSL

## Тесты

- `files-find` — рекурсивный обход файлов
- `files-create-delete` — создание и удаление 10 000 маленьких файлов
- `npm-install` — установка npm-зависимостей из `package-lock.json` с кешем и без кеша

## Корректность тестов

- Наборы тестов Bash и PowerShell используют максимально эквивалентные операции для своих runtime и системных API.
- В Windows используется MSYS2 Bash, в WSL используется нативный WSL Bash.
- Тесты запускаются на свободной системе
- Тесты запускается по очереди, параллельный запуск запрещен
- Тесты можно запустить несколько раз, все результаты будут сохраняться в CSV

## Режимы

- `native` — работа с файловой системой текущего окружения
- `proxy` — работа через границу Windows ↔ WSL

## Настройки

- Настройки по умолчанию лежат в `.env`.
- Все скрипты автоматически подхватывают `.env`.

## Запуск тестов

Результаты тестов сохраняются в `results/results.csv` и `results.txt`.

Файлы не очищаются автоматически, чтобы обеспечить возможность многократных запусков и накопления статистики для
последующего анализа.

PowerShell:

```powershell
.\run.ps1
```

Bash:

```bash
# настроить доступ один раз
sudo chmod +x *.sh
sudo chmod +x bash/*.sh

./run.sh
```

### Запуск одного теста

PowerShell:

```powershell
.\powershell\npm-install.ps1 $false $true
```

Bash:

```bash
# настроить доступ один раз
sudo chmod +x *.sh bash/*.sh

bash/npm-install.sh false true
```

## Как читать результаты

### Внутри одного окружения

| Источни запуска | Сравнение                                                     | Что показывает                                                                       |
|-----------------|---------------------------------------------------------------|--------------------------------------------------------------------------------------|
| Windows         | PowerShell `native` vs PowerShell `proxy`                     | Производительность PowerShell при доступе к файлам WSL через границу Windows ↔ WSL   |
| Windows         | MSYS2 Bash (Windows) `native` vs MSYS2 Bash (Windows) `proxy` | Производительность MSYS2 Bash (Windows) при доступе к файлам WSL                     |
| WSL2            | WSL Bash `native` vs WSL Bash `proxy`                         | Производительность WSL Bash при доступе к файлам Windows через границу Windows ↔ WSL |

### Между окружениями

| Запуск         | Сравнение                                          | Что показывает                                                   |
|----------------|----------------------------------------------------|------------------------------------------------------------------|
| Windows / WSL2 | MSYS2 Bash (Windows) `native` vs WSL Bash `native` | Суммарную разницу файловой системы и runtime между Windows и WSL |

Сравнения между разными окружениями включают и разницу файловой системы, и разницу runtime.
